The WireGuard server configuration lives in `/etc/wireguard/wg0.conf`. The install script generates this file automatically, but understanding its structure is essential for adding peers, updating firewall rules, and troubleshooting connectivity.

## wg0.conf structure

The file has two sections: an `[Interface]` block that describes the server itself, and one `[Peer]` block per connected client.

```ini
[Interface]
Address = 10.0.0.1/24
ListenPort = 51820
PrivateKey = <CLAVE_PRIVADA_SERVIDOR>

PostUp = iptables -A FORWARD -i wg0 -j ACCEPT; iptables -A FORWARD -o wg0 -j ACCEPT; iptables -t nat -A POSTROUTING -o <INTERFAZ_RED> -j MASQUERADE
PostDown = iptables -D FORWARD -i wg0 -j ACCEPT; iptables -D FORWARD -o wg0 -j ACCEPT; iptables -t nat -D POSTROUTING -o <INTERFAZ_RED> -j MASQUERADE

# Cliente 1
[Peer]
PublicKey = <CLAVE_PUBLICA_CLIENTE_1>
AllowedIPs = 10.0.0.2/32

# Cliente 2
[Peer]
PublicKey = <CLAVE_PUBLICA_CLIENTE_2>
AllowedIPs = 10.0.0.3/32

# VPS WAF / Apache
[Peer]
PublicKey = <CLAVE_PUBLICA_CLIENTE_3>
AllowedIPs = 10.0.0.10/32
```

## Key parameters

### Interface block

| Parameter | Value | Description |
|---|---|---|
| `Address` | `10.0.0.1/24` | VPN IP of the server and subnet definition |
| `ListenPort` | `51820` | UDP port WireGuard listens on |
| `PrivateKey` | server private key | Generated by the install script at `/etc/wireguard/server_private.key` |
| `PostUp` | iptables commands | Runs when the interface comes up |
| `PostDown` | iptables commands | Runs when the interface goes down |

### Peer block

| Parameter | Value | Description |
|---|---|---|
| `PublicKey` | client public key | The client's WireGuard public key — case-sensitive |
| `AllowedIPs` | e.g. `10.0.0.2/32` | IP address the server will route to this peer — use `/32` for a single host |

## How PostUp and PostDown work

The `PostUp` rules enable NAT so that VPN clients can reach the internet through the server:

```bash
# Allow forwarding between wg0 and the physical interface
iptables -A FORWARD -i wg0 -j ACCEPT
iptables -A FORWARD -o wg0 -j ACCEPT

# Masquerade outbound traffic on the physical interface
iptables -t nat -A POSTROUTING -o <INTERFAZ_RED> -j MASQUERADE
```

`PostDown` removes these rules when WireGuard stops, preventing stale iptables entries:

```bash
iptables -D FORWARD -i wg0 -j ACCEPT
iptables -D FORWARD -o wg0 -j ACCEPT
iptables -t nat -D POSTROUTING -o <INTERFAZ_RED> -j MASQUERADE
```

<Warning>
  The interface name (`<INTERFAZ_RED>`) must match your server's actual network interface. Using the wrong interface name means traffic will be accepted into the VPN but silently dropped rather than routed onward.
</Warning>

## Detecting the correct network interface

The install script detects the interface automatically, but if you edit `wg0.conf` manually, confirm the interface name first:

```bash
ip route | grep default
```

Example output:

```
default via 212.x.x.1 dev ens6 proto static
```

In this example, the interface is `ens6`. Common names include `eth0`, `ens6`, and `enp3s0` — they vary by cloud provider and kernel version. Update `wg0.conf` accordingly if your interface differs.

## VPN subnet and IP assignment

The server uses the `10.0.0.0/24` subnet:

| Address | Role |
|---|---|
| `10.0.0.1` | Server (VPN gateway) |
| `10.0.0.2` | First client |
| `10.0.0.3` | Second client |
| `10.0.0.10` | Reserved for infrastructure peers (e.g. WAF, Apache VPS) |

Assign each new client a unique `/32` address within `10.0.0.0/24`. The management panel and `add-client.sh` handle this automatically.

## Adding peers

<Tabs>
  <Tab title="Management panel (recommended)">
    Open the panel at `http://YOUR_VPS_IP:8443`, click **Add client**, and fill in the client name. The panel generates a key pair, assigns the next available IP, writes the `[Peer]` block to `wg0.conf`, and reloads WireGuard — all without a full restart.
  </Tab>
  <Tab title="add-client.sh">
    Use this script when you already have the client's public key:

    ```bash
    sudo bash onboarding/add-client.sh <name> <public_key> <assigned_ip>
    ```

    Example:

    ```bash
    sudo bash onboarding/add-client.sh alice CMvCw4gSxdv5xwkgx...= 10.0.0.2
    ```

    The script registers the peer live with `wg set`, appends the block to `wg0.conf`, and saves with `wg-quick save wg0`.
  </Tab>
  <Tab title="Manual">
    Append a `[Peer]` block directly to `/etc/wireguard/wg0.conf`:

    ```ini
    # Cliente: <name>
    [Peer]
    PublicKey = <key>
    AllowedIPs = <ip>/32
    ```

    Then reload (see below). Manual edits are useful for infrastructure peers with fixed IPs.
  </Tab>
</Tabs>

## Reloading configuration

Choose the method that fits the change you're making:

### Hot-reload with `wg set` (no downtime)

Use this to add or remove a peer without interrupting existing connections:

```bash
# Add a peer
sudo wg set wg0 peer <CLIENT_PUBLIC_KEY> allowed-ips <CLIENT_IP>/32

# Remove a peer
sudo wg set wg0 peer <CLIENT_PUBLIC_KEY> remove

# Persist the change to wg0.conf
sudo wg-quick save wg0
```

### Full restart with wg-quick

Required after editing `[Interface]` parameters such as `Address`, `ListenPort`, or `PostUp`/`PostDown`:

```bash
sudo wg-quick down wg0 && sudo wg-quick up wg0
```

<Note>
  A full restart briefly drops all active VPN connections. Clients reconnect automatically within a few seconds.
</Note>

## Persistence across reboots

The install script already enables the systemd service. To verify or re-enable it:

```bash
sudo systemctl enable wg-quick@wg0
sudo systemctl status wg-quick@wg0
```

To check whether WireGuard will start on next boot:

```bash
sudo systemctl is-enabled wg-quick@wg0
# Expected output: enabled
```
